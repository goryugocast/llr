# Actionable_Data_Extraction: 行動履歴データの抽出と統合

## 1. 目的（Objective）
プロジェクト「Actionable Data」において、過去の「たすくま」記録（mbox および Evernote）から、AIが文脈や傾向を理解するための「長期記憶資産」を再現性高く抽出・統合するための仕様を定義する。

## 2. 出力データ構造（Target Schema: JSONL）
各行動ログは以下の JSON オブジェクトとして 1 行 (JSONL) で表現される。情報の最大化を目的とし、ログには「タスク（task）」と「デイリーサマリー（daily_summary）」の2種類が存在する。

```json
// Task エントリ
{
  "type": "task",
  "date": "YYYY-MM-DD",     // 日付
  "emoji": "🎨",            // タスク名の先頭絵文字
  "task": "タスク名",       // タスク名
  "project": "プロジェクト", // 紐づくプロジェクト属性
  "start": "HH:mm",         // 開始時刻
  "end": "HH:mm",           // 終了時刻
  "tags": ["タグ1", ...],   // 全ソースから結合されたカテゴリタグ
  "memo": "詳細内容"         // 全ソースから結合された詳細メモ（複数行可）
}

// Daily Summary エントリ
{
  "type": "daily_summary",
  "date": "YYYY-MM-DD",     // 日付
  "title": "ノートタイトル",
  "narrative": "...",       // その日の総括や日記的なテキスト
  "tags": ["タグ1", ...],   
  "raw_text": "..."         // 抽出元の完全な生テキスト（情報の不可逆な欠落防止用）
}
```

## 3. ソースデータの構造解析と抽出ロジック

### 3.1. たすくま mbox ログ
- **構造**: `mailbox.mbox` 形式。各メールは一日のログサマリー。
- **抽出ポイント**:
  - **本文 (text/plain)**: `Memo` セクションに全情報が集約されている。
  - **タスク見出し**: `(HH:mm - HH:mm)` が続く行の直前。
  - **タグ情報**: `(HH:mm - HH:mm)` の直後の文字列。
  - **解析ロジック**: 正規表現を用いて、テキストベースでタスク・時間・タグ・メモを分離する。

### 3.2. Evernote (.enex) ログ
- **構造**: XML (`<note>` タグの集合)。
- **抽出ポイント**:
  - **タイトル**: `YYMMDD` 形式の日付が含まれる。
  - **作成日時 (`created`)**: 日付がタイトルにない場合のフォールバック。
  - **本文 (`content`)**: HTML 表形式でタスクがリスト化されている。
  - **メモ**: `h4` 見出しおよびそれに続く `div`/`p`/テキストノード。
- **解析ロジック**: `xml.etree.ElementTree.iterparse` によるストリーミング解析。大容量（数GB）に対応するため、メモリ効率を最優先する。

## 4. 統合・クリーンアップ原則（情報の和集合・Appendマージ）
「Aのデータを正としてBを上書きする」のではなく、「すべてのソースから情報を一滴も漏らさずに結合する」概念で統合する。
1. **ユニークキー**: `(date, start, task)` をキーとして同一タスクを同定。
2. **情報の合流**: 同じキーの記録が複数ソース（SQLiteとEvernote等）にある場合、`memo` や `tags` は「改行で連結」「配列を結合」してすべて保持する。
3. **時系列ソート**: 最終出力は `date` > `type (daily_summary優先)` > `start` の順でソートされる。

## 5. 再現手順とファイル構成（Reproducibility & Files）
1. 各ソースに対応した抽出スクリプト（`extract_mbox.py` / `extract_enex.py` / `extract_sqlite.py`）を実行。
2. 中間ファイルとしてソースごとの JSONL を `tc/archive_jsonl/` 等に生成（現在は整理のためアーカイブ済み）。
3. `merge_history.py` により最終的な統合マスターデータ **`tc/history.jsonl`** を作成する。このファイルが唯一の「正」となる全知全能のデータ元である。

## 6. 標準化キー（Standard Keys）の継続的運用ルール（Progressive Refinement）
データスキーマV2における `standard_project` および `standard_task` の付与は、単発の変換処理ではなく、AIによる継続的な監視と統合のプロセスとして定義する。

**【AIの基本行動原則】**
1.  **大局観の監視**: AIは常に `standard_task` の全体像を見張り、バラバラな記録の統合や、過剰にまとめられた記録の分割が可能かどうかを考え続ける。
2.  **ラフな初期割り当て**: 最初から100%の完全なマッピングを目指さない。まずは明白な反復行動（睡眠、食事、入浴など）にざっとパラメーターを付ける。
3.  **例外の許容と追跡**: 判定不能なものや極めて例外的なものは、無理に既存キーに押し込めず「空欄」または「例外キー（例: `exception_xxx`）」をつけておく。
4.  **漸進的統合（Progressive Integration）**: 空欄や例外キーの項目を少しずつ分析する。一定のパターンが見い出せた場合、既存のキーに組み込めるか、あるいは新設スべきかを検討し、統合を実行する。

この運用ルールにより、10年分の表記揺れデータを「生きたシステム」として徐々に洗練（リファイン）させていく。
